The Big 3 Display Property Values: Block, Inline, & Inline-block

Friday Jan 24, 2015

What does it mean to display inline vs inline blocks?

In my experience in CSS thus far, I have used the "display" property very, very often. I am sur emost of you have as well. However, despite how much I use it, I realized that I never had a clear understanding of what each of the three manin values for this property affected the layout of my web pages. If one didn't produce the desired layout change, I would simply try one of the others. If you are like me, hopefully this will help better your understanding so you can save time by picking the right display value the first time, every time.

When it comes to block, inline, and inline-block, the main differences are in how the width and height are set, whether or not you can change the width and height, and if it can have neighbors (only one of them can't) We'll start with block. you write it like so:

display: block;
What you need to know about blocks:
1. width is is set by its parent and its height set by its content.
2. You CAN change the widths and heights.
3. This DOES NOT ALLOW horizontal neighbors--This means if you use a block, the content inside that block will be the only content on that line.

Next up, inline. Written as: display: inline;
What you need to know about inline:
1. An element set to Inline will have its width and height determined by its content.
2. You cannot change the height and width. It will simply be the size of the content you put in.

Last, inline-block. Written as: display: inline-block;
What you need to know about inline-block:
1. It's the same as block
**EXCEPT**
2. Width is determined by its content.
3. BUT, you CAN set its width and height.

Blog #1

Version Control, Git & Github

Friday Jan 16, 2015

What are the benefits of Version Control?

To better understand Version Control, I like to imagine a Word document that only allows you to save documents as new documents, so you can never save over a document you already saved. Editing a paper you wrote and saved as "Paper"? Well you have to save your edit as "Paper-Edit-1". Of course, version control does allow you to keep and update a master version, but it keeps track of all your edits in much the same way as my twisted example above. The benefits of this are that you (1) never lose previous work in case you royally screw up your project and (2) can work with many different people in many different places at the same time and not step on eachothers toes. Elaborating on benefit (2),this means that more than one person can be working on that paper that's due at midnight at the same time and your version control system will lump the two edits together into a beatufil master document as long as you didn't changes the same parts!

How does git help you keep track of changes?

Git keeps track of changes through the 3 levels of working, tracked, and committed. So you work on your document locally on your own computer and save it. According to git, you're document is now in the working stage. All you have to do it add it to git via "git add " and it is now in the tracked stage. Review your work one more time to make sure you are satisfied with your update and type in "git commit" and there you go! Your changes are now saved as a commit. Feel free to type in "git push" and your new update will be pushed to your remote space (such as Github).

Why use Github to store your code?

Github gives you a place to push your commits to a remote place where it is accessible by anyone (with access to your profile) on any computer (maybe you're at a different computer). It shows your master code, all the branches, and all the commits. It also includes a social aspect that allows you to connect with friends, colleagues, and even strangers to share and write code. It seems like a great environment for any developer.
